# ADR-001 — Adopter un verrou pessimiste PostgreSQL pour l'allocation de la dernière place

**Statut** : Accepté
**Date** : 2026-04-29
**Auteurs** : Samy Abdelmalek, Tristan Sanjuan, Saam

## Contexte

Le `TicketModule` doit allouer des billets sur des événements à jauge limitée. Sur les événements populaires (release de billetterie à `T0`), plusieurs centaines de requêtes simultanées convergent sur les dernières places. Si deux requêtes parallèles voient `seatsReserved < capacity` puis incrémentent toutes les deux le compteur, la jauge est dépassée — survente interdite par les règles métier.

Forces en présence :
- **Cohérence forte requise** : zéro survente tolérée (contrainte métier dure, communiquée aux organisateurs).
- **Pic de contention bref** : la concurrence est extrême sur ~500ms à l'ouverture, faible le reste du temps.
- **Stack** : PostgreSQL 15 unique (pas de sharding), Node.js stateless derrière un load balancer.
- **UX** : un client qui obtient `409 Sold Out` doit le savoir immédiatement, pas après 3 retries silencieux.

## Décision

Nous utilisons un verrou pessimiste `SELECT ... FOR UPDATE` sur la ligne `event` au sein de la transaction d'allocation du ticket. La requête concurrente perdante reçoit `409 Sold Out` après attente bornée (`lock_timeout = 5s`).

## Alternatives envisagées

### Verrou optimiste avec retry (compare-and-swap sur `version`)
Chaque transaction lit la version courante, tente l'`UPDATE WHERE version = X`, et retry sur conflit. Élégant et sans contention de lock, mais sur le cas d'usage « dernière place », le taux d'échec à la première tentative explose : avec 200 requêtes pour 1 place, 199 doivent retry. Le retry storm dégrade la latence p99 plus que le verrou pessimiste, et complexifie le code applicatif. Écarté car la fréquence de la contention extrême (releases à `T0`) est précisément le scénario où l'optimiste est le pire.

### File de réservation asynchrone (queue → worker unique)
Toutes les demandes sont enfilées dans une queue RabbitMQ traitée séquentiellement par un worker unique par événement. Garantit absence de survente sans verrou DB. Écarté pour deux raisons : (1) latence utilisateur dégradée (réponse asynchrone "vous êtes 142e dans la file" — UX pire que `409`), (2) point de défaillance unique par événement, (3) complexité de gestion des timeouts et abandons.

### Verrou applicatif distribué via Redis (Redlock)
Mutex distribué Redis avant l'opération SQL. Ajoute une dépendance critique (Redis devient single-point-of-failure pour la billetterie) et duplique la garantie de cohérence que PostgreSQL fournit déjà nativement. Écarté : on ne paie pas la complexité d'un consensus distribué quand un seul nœud SQL fait l'autorité.

## Conséquences

### Positives
- Zéro survente garantie par PostgreSQL, pas par du code applicatif.
- Implémentation triviale (5 lignes de SQL), peu de surface de bug.
- Pas de dépendance supplémentaire (Redis, queue dédiée).
- Le `409 Sold Out` est immédiat et explicite côté client.

### Négatives
- Sérialisation effective des transactions concurrentes sur la même ligne `event` : la latence p99 sur la dernière place peut atteindre `lock_timeout` (5s).
- Si un worker crash en pleine transaction, le lock est libéré au timeout PostgreSQL (par défaut), mais une mauvaise configuration peut bloquer toutes les requêtes suivantes. Surveillance requise sur `pg_stat_activity` (locks en attente).
- Couple le `TicketModule` à PostgreSQL : un futur passage à un store distribué (Cassandra, Spanner) demanderait une refonte de la stratégie de concurrence.

### Dette acceptée
La scalabilité horizontale de la billetterie est plafonnée par le débit d'écriture d'un seul nœud PostgreSQL primaire. Pour les événements à plus de ~10 000 places ouvertes simultanément avec contention massive, il faudra réévaluer (sharding par événement, partitioning, ou bascule vers une file de réservation). Aucun événement de cette taille n'est prévu sur SupEvents à 12 mois.
