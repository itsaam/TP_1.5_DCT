# ADR-002 — Appliquer un retry exponentiel borné avec bascule en DLQ pour les événements asynchrones

**Statut** : Accepté
**Date** : 2026-04-29
**Auteurs** : Samy Abdelmalek, Tristan Sanjuan, Saam

## Contexte

Les modules consommateurs d'événements RabbitMQ (`NotificationModule`, `PaymentModule`, `TicketModule`) sont sensibles à des défaillances transitoires de dépendances externes : SendGrid 5xx/429, webhook Stripe en rejeu, base de données momentanément indisponible. Une politique de gestion des erreurs uniforme est nécessaire pour :
- éviter la perte silencieuse de messages (un mail de confirmation jamais envoyé après paiement = incident métier majeur) ;
- éviter le retry infini qui sature le broker et masque les pannes durables ;
- offrir un point d'observation et de rejeu manuel pour les opérateurs.

Forces en présence : SLA d'engagement utilisateur (mail de confirmation < 1 min en nominal), volumétrie modérée (~10k notifs/jour), équipe ops réduite (rejeu manuel possible mais pas à grande échelle).

## Décision

Tous les consommateurs d'événements appliquent la politique suivante :
- **Erreur transitoire** (5xx, 429, timeout, connexion refusée) : retry avec backoff exponentiel, base 30s, facteur 4, plafond 5 tentatives (cumul ≈ 2h40).
- **Erreur permanente** (4xx hors 429, erreur de désérialisation, contrainte métier violée) : pas de retry, bascule directe en DLQ.
- **Épuisement du retry** : message envoyé sur la DLQ dédiée par exchange (`<exchange>.dlq`), événement `*.failed.permanent` publié pour observabilité.

## Alternatives envisagées

### Retry infini avec backoff plafonné
Continuer à retenter à intervalle régulier (ex. toutes les heures) jusqu'à succès, sans DLQ. Écarté : un message « zombie » consommant des ressources indéfiniment masque les vraies pannes (l'alerte n'arrive jamais), et un message corrompu ou métier-incohérent ne deviendra jamais valide.

### DLQ immédiate sans retry
Toute erreur bascule en DLQ, le rejeu est manuel. Trop strict : un blip réseau de 2 secondes vers SendGrid déclencherait un rejeu manuel pour des centaines de mails, ce qui ne passe pas à l'échelle pour notre équipe ops.

### Circuit breaker sur l'adapter sortant
Plutôt qu'un retry par message, ouvrir un circuit breaker global sur SendGrid quand le taux d'erreur dépasse un seuil. Complémentaire mais pas exclusif : le circuit breaker protège les ressources, mais ne résout pas le sort des messages individuels. Reporté à une itération ultérieure (combinable avec la décision actuelle).

## Conséquences

### Positives
- Politique uniforme : un seul runbook ops pour rejouer une DLQ, quel que soit le module.
- Borne supérieure connue sur la persistance d'un message non livré (≈ 2h40), exploitable pour les SLA.
- Distinction explicite transitoire/permanent évite les retries inutiles sur des erreurs déterministes (email invalide).
- Observabilité directe : la profondeur de la DLQ est une métrique d'alerte naturelle.

### Négatives
- La latence d'une notification dégradée peut atteindre 2h40 avant abandon, ce qui est long pour l'utilisateur final même si le mail finit par partir.
- Nécessite la mise en place d'un mécanisme de delayed messages (plugin `rabbitmq_delayed_message_exchange` ou pattern TTL + DLX) — complexité opérationnelle additionnelle sur le broker.
- Le rejeu manuel depuis la DLQ reste une charge ops si les pannes sont fréquentes.

### Dette acceptée
La classification transitoire/permanent est codée en dur dans chaque adapter et non centralisée. Si un nouveau code d'erreur SendGrid apparaît, il sera traité comme transitoire par défaut (potentiellement à tort) jusqu'à ce qu'on l'ajoute explicitement à la liste des permanents. Un audit trimestriel de la DLQ permettra de remonter ces cas.
