# ADR-003 — Émettre un JWT applicatif court doublé d'un refresh token stateful en Redis

**Statut** : Accepté
**Date** : 2026-04-29
**Auteurs** : Samy Abdelmalek, Tristan Sanjuan, Saam

## Contexte

Après l'authentification OIDC via le SSO école, l'`AuthModule` doit émettre un jeton de session applicatif consommé par tous les autres modules pour identifier l'utilisateur et vérifier ses rôles. Plusieurs forces en tension :

- **Performance** : la vérification du jeton est sur le chemin chaud de chaque requête API ; un appel réseau systématique (Redis, base) à chaque requête coûte cher.
- **Révocabilité** : un compte désactivé côté SSO école (étudiant qui quitte l'établissement) doit pouvoir être déconnecté rapidement, idéalement sous 15 minutes.
- **Sécurité** : exposition au vol de jeton (XSS, fuite de logs) — la fenêtre d'exploitation doit être bornée.
- **Multi-instance** : le backend est stateless derrière un load balancer ; pas de session sticky.

## Décision

Nous émettons un **JWT applicatif signé (HS256, secret côté serveur) à durée de vie courte (15 minutes)**, vérifiable sans appel réseau, doublé d'un **refresh token opaque (UUID v4) stocké en Redis** avec une TTL de 30 jours, révocable individuellement. Le `jti` du JWT peut être ajouté à une blacklist Redis (TTL = durée de vie restante) pour révocation immédiate exceptionnelle.

## Alternatives envisagées

### Sessions serveur classiques (cookie de session opaque + lookup Redis à chaque requête)
Approche sans JWT : cookie d'identifiant aléatoire, données de session côté Redis. Simple et révocable instantanément. Écarté en raison de l'appel Redis sur le chemin chaud de chaque requête (~0.5ms de latence × débit nominal = pression Redis significative), et d'un couplage opérationnel fort à Redis (panne Redis = panne d'auth complète, alors qu'avec JWT court, l'auth survit jusqu'à 15 min).

### JWT stateless pur (long, sans refresh, sans blacklist)
Un seul JWT longue durée (24h ou plus), aucun stockage côté serveur. Maximale en performance mais impossible à révoquer avant expiration — un jeton volé reste valide 24h, un compte désactivé reste actif 24h. Inacceptable au regard du contexte SSO école (révocation rapide attendue par l'établissement).

### Tokens d'accès courts portés par OIDC seul (introspection à chaque requête)
Réutiliser directement les `access_token` du SSO école et appeler `/introspect` ou `/userinfo` à chaque requête. Couple le système au SSO école (panne SSO = API inutilisable), et soumet l'API au rate-limit du SSO école qui n'est pas dimensionné pour notre trafic applicatif. Écarté.

## Conséquences

### Positives
- Vérification du JWT sans I/O (signature locale) → latence négligeable sur le chemin chaud.
- Fenêtre d'exposition d'un jeton volé bornée à 15 min, sauf si l'attaquant possède aussi le refresh (cookie `HttpOnly` distinct).
- Révocation effective sous 15 min via simple non-renouvellement, ou immédiate via blacklist Redis pour les cas critiques.
- Pas de dépendance Redis sur le chemin chaud de validation — Redis n'intervient qu'au refresh (toutes les 15 min) et lors d'événements de révocation.

### Négatives
- Complexité accrue côté client : gérer le refresh automatique sur `401`, éviter les races (deux requêtes concurrentes qui rafraîchissent simultanément).
- Deux secrets à gérer en sécurité (signature JWT + chiffrement des refresh tokens en Redis), avec rotation à orchestrer.
- La blacklist Redis introduit un couplage Redis pour les révocations immédiates — si Redis tombe pendant une révocation critique, la fenêtre redevient 15 min. Acceptable au regard du périmètre SupEvents.

### Dette acceptée
La rotation du secret de signature JWT impose une fenêtre de double-validation (ancien + nouveau secret) durant la rotation, qui n'est pas encore automatisée. La rotation manuelle reste un risque opérationnel à l'horizon où l'application aura besoin de tourner les secrets régulièrement. À industrialiser au prochain cycle de hardening sécurité.
